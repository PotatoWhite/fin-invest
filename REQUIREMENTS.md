# 투자 모니터링 시스템 v2 — 요구사항 정의서

## 배경

기존 potato-fin 시스템의 한계를 극복하기 위한 재구축.

### potato-fin의 한계
- 수동적 구조: 데이터 수집 → 프롬프트에 일괄 삽입 → Claude 1회 호출
- 한국주식 펀더멘탈 API 부재 (yfinance 의존)
- 차트 데이터 축적 없음 (일봉 기반 기술분석만)
- 예측 추적이 JSON 파일 기반 (구조화 부족)
- 자기개선이 코드 수정 수준 (프롬프트 편집)
- 단일 에이전트 분석 (분야별 깊이 부족)
- 신호 품질 검증 없음 (거짓 신호 필터링 부재)
- 인과 사슬 추적 없음 (이벤트 간 연결 부재)
- 비트코인/암호화폐 분석 없음

## 2. 신호 처리 시스템

### 2.1 원칙
> 신호를 찾는 것보다 **거짓 신호를 걸러내는 것**이 더 중요하다.

모든 신호는 분석에 사용되기 전에 **품질 검증**을 통과해야 한다.

### 2.2 7단계 거짓 신호 필터

| 단계 | 필터 | 판정 기준 | 미통과 시 |
|------|------|----------|----------|
| 1 | **거래량 확인** | 가격 변동 대비 거래량이 평균 이상인가 | 품질 ×0.3 |
| 2 | **시장폭 확인** | 인덱스 상승 시 상승종목 > 하락종목인가 | 품질 ×0.5 |
| 3 | **유동성 필터** | 정규장/프리마켓/점심시간/공휴일 전 | 품질 ×0.3~0.7 |
| 4 | **뉴스 중복 제거** | 같은 뉴스의 반복 보도 여부 | 2번째 ×0.3, 3번째 ×0.1 |
| 5 | **스톱헌팅 탐지** | 지지선 이탈 후 5분 내 회복 | 하락신호 무효화 |
| 6 | **교차자산 확인** | 채권/VIX/환율/금/폴리마켓이 동의하는가 | 2/5 미만 동의 시 ×0.5 |
| 7 | **역사적 검증** | 유사 조건에서 이 신호의 과거 적중률 | 적중률 <50% 시 ×0.3 |

```
최종 품질 = Π(각 필터 점수)
  > 0.7  → 신뢰, 전체 가중치로 분석
  0.4~0.7 → 참고만, 가중치 50% 할인
  < 0.4  → 무시 (거짓 신호)
```

### 2.3 신호 품질 DB

```sql
signal_quality (
    signal_id, timestamp, ticker,
    signal_type,          -- 'price_move', 'volume_spike', 'news', 'technical_breakout'
    raw_magnitude,        -- 원시 신호 강도
    volume_score,         -- Filter 1
    breadth_score,        -- Filter 2
    liquidity_score,      -- Filter 3
    dedup_score,          -- Filter 4
    stophunt_score,       -- Filter 5
    crossasset_score,     -- Filter 6
    historical_score,     -- Filter 7
    final_quality,        -- 최종 품질 점수
    was_real INTEGER      -- 나중에 검증: 진짜였나 가짜였나
)
```

## 3. 인과 사슬 추적 및 수치화

### 3.1 뉴스 반감기 기준표 (재검토)

반감기는 **레짐에 따라 달라짐**. Risk-Off 시 뉴스 반감기가 길어지고 (공포 지속), Risk-On 시 짧아짐 (빨리 잊음).

| 뉴스 유형 | 초기 충격 | 반감기 (Risk-On) | 반감기 (Risk-Off) | 완전 흡수 | 감쇠 형태 |
|-----------|----------|-----------------|------------------|----------|----------|
| **실적 서프라이즈** | ±3-8% | 5-10일 | 7-14일 | 60거래일 | 지수 + PEAD 잔류 |
| **FOMC 금리결정** | ±0.5-2% | 2-4시간 | 4-8시간 | 1-2일 | 스텝 (사전선반영 80%) |
| **CPI/고용 서프라이즈** | ±0.3-1.5% | 4-8시간 | 8-16시간 | 1-3일 | 지수 |
| **지정학 (전쟁, 제재)** | ±1-5% | 2-6시간 | 1-3일 | 1-3일(R-On), 수주(R-Off) | 지수 + 불확실성 프리미엄 잔류 |
| **CEO 교체/M&A** | ±3-15% | 3-5일 | 5-7일 | 2-4주 | 지수 |
| **관세/무역 정책** | ±1-4% | 1-3일 | 3-7일 | 1-2주 | 스텝 (시행일까지 유지) |
| **SNS/밈/루머** | ±2-10% | 1-4시간 | 2-6시간 | 1일 | 급속 지수 |
| **규제/법률 변경** | ±2-8% | 3-7일 | 7-14일 | 수주-수개월 | 이중 감쇠 (초기 급감 + 장기 잔류) |
| **중앙은행 정책 변경** | ±1-3% | 1-2일 | 2-5일 | 1-3주 | 스텝 + 지수 |
| **자연재해/팬데믹** | ±2-10% | 3-7일 | 1-3주 | 수개월 | 장기 잔류 |

**감쇠 형태 유형:**
- **지수(exponential)**: 표준 `e^(-λt)` 감쇠
- **스텝(step)**: 특정 시점까지 유지 후 급감 (ex: FOMC는 다음 FOMC까지 부분 유지)
- **이중 감쇠**: 초기 30% 급감 + 나머지 70%는 천천히 감쇠
- **잔류 효과**: 반감기 이후에도 불확실성 프리미엄이 남음 (지정학, 규제)

> 이 값들은 초기값이며, 자기개선 시스템이 실측 데이터로 **레짐별로 지속 보정**.

**파라미터 저장: DB `model_params` 테이블**
```sql
model_params (
    id INTEGER PRIMARY KEY,
    category TEXT NOT NULL,       -- 'half_life', 'impact_magnitude', 'regime', 'signal_filter'
    param_name TEXT NOT NULL,     -- 'earnings_surprise_halflife_risk_on'
    value REAL NOT NULL,          -- 120 (시간)
    calibration_factor REAL DEFAULT 1.0,  -- 자기개선 보정계수
    updated_by TEXT,              -- 'initial', 'improvement_agent', 'manual'
    updated_at TEXT DEFAULT (datetime('now','localtime')),
    UNIQUE(category, param_name)
)
```
개선 에이전트가 이 테이블의 값을 수정 → 코드 변경 없이 파라미터 조정 가능.

### 3.2 충격 크기 모델 (Impact Magnitude)

같은 유형의 뉴스라도 충격 크기가 다르다. "CPI 0.1% 상회"와 "CPI 1.0% 상회"는 완전히 다른 반응.

**충격 크기 공식:**
```
Impact = BaseMagnitude(event_type)
       × SurpriseFactor(actual, expected)
       × RegimeSensitivity(current_regime)
       × PositioningAmplifier(current_positioning)
       × LiquidityFactor(current_liquidity)
```

| 요소 | 범위 | 설명 |
|------|------|------|
| **BaseMagnitude** | 뉴스 유형별 | 반감기 테이블의 "초기 충격" 기본값 |
| **SurpriseFactor** | 0~5x | `|실제값 - 시장 기대값|`에 비례. 예상과 일치하면 0 (이미 선반영) |
| **RegimeSensitivity** | 0.5~2.0x | Risk-On 0.5 (둔감), Risk-Off 2.0 (과민), 전환기 1.5 |
| **PositioningAmplifier** | 0.5~3.0x | 모두 롱인데 악재→패닉매도 3.0x, 중립이면 1.0x |
| **LiquidityFactor** | 0.5~2.0x | 유동성 낮으면 증폭, 풍부하면 흡수 |

**핵심: 시장이 이미 아는 건 뉴스가 아니다. 충격 = 예상과의 괴리.**

```
예시: CPI 발표
  Base: ±1.0% (경제지표 기본)
  Surprise: 예상 3.0%, 실제 3.5% → 서프라이즈 0.5%p → ×2.5
  Regime: VIX 28 (Risk-Off) → ×1.8
  Positioning: 시장 롱 포지션 과다 → ×1.5
  Liquidity: 정규장, 정상 → ×1.0

  최종 충격: 1.0% × 2.5 × 1.8 × 1.5 × 1.0 = -6.75%
```

자기개선 시스템이 각 승수의 정확도를 추적하고 보정.

### 3.3 인과 링크 수치화 (Causal Link)

모든 이벤트 → 영향 관계를 정량화:
```
CausalLink {
    event       → 원인 이벤트
    target      → 영향 받는 대상
    chain_depth → 1차, 2차, 3차
    magnitude   → 예상 충격 (%)
    delay       → 전파 지연 (시간)
    half_life   → 효과 반감기 (시간)
    confidence  → 이 링크의 신뢰도 (0-1)
    chain_confidence → 누적 신뢰도 (사슬 전체 곱)
}
```

신뢰도 감쇠: 사슬이 길어질수록 곱연산
```
1차: 0.90 → 2차: 0.90×0.85=0.765 → 3차: 0.497 → 4차 이상: 노이즈
```
**3차까지만 예측에 활용.**

### 3.4 잔여 영향 계산

현재 시점에서 모든 활성 이벤트의 감쇠 효과 합산:
```
잔여영향(t) = magnitude × e^(-0.693 × elapsed / half_life) × confidence
순영향 = Σ(양의 잔여영향) + Σ(음의 잔여영향)
```

반감기 3배 이상 경과 → 무시 (잔여 12.5% 미만).

### 3.5 시장 영향 3-Layer 모델

뉴스 반감기(Layer 1)만으로는 전쟁, 정치 불안 같은 **지속적 리스크**를 반영할 수 없다.
3개 레이어를 합산하여 총 시장 영향을 계산.

```
총 영향 = Layer 1 (이벤트 충격) + Layer 2 (지속적 리스크 프리미엄) + Layer 3 (폴리마켓 확률 반영)
```

#### Layer 1: 뉴스 이벤트 (반감기 모델)
일회성 이벤트의 충격과 감쇠.
```
"이란이 이스라엘에 미사일 발사" → 충격 -3% → 반감기 6시간 → 감쇠
```
위의 3.1~3.4 섹션이 이 레이어.

#### Layer 2: 지속적 리스크 상태 (Geopolitical Risk Register)
전쟁, 정치 불안, 무역전쟁 등 **진행 중인 리스크**는 반감기가 아닌 **상태**로 추적.
해소될 때까지 리스크 프리미엄이 유지됨.

```sql
geopolitical_risks (
    id INTEGER PRIMARY KEY,
    name TEXT,                    -- '이란-이스라엘 분쟁', '한국 탄핵 정국'
    category TEXT,                -- 'war', 'political', 'trade', 'sanctions', 'social'
    status TEXT,                  -- 'escalating', 'stable', 'de_escalating', 'resolved'
    severity INTEGER,             -- 1-10 (10=세계대전급)

    -- 시장 영향 파라미터
    risk_premium_json TEXT,       -- {"WTI": +8.0, "gold": +3.0, "KOSPI": -2.0, "USD/KRW": +2.0}
    affected_assets TEXT,         -- JSON: ["WTI", "gold", "USD/KRW", "방산주"]
    safe_haven_boost REAL,        -- 안전자산 프리미엄 (%)

    -- 확률 시나리오
    scenarios_json TEXT,          -- [{"name":"확대","prob":0.30,"impact":{...}}, ...]
    escalation_prob REAL,         -- 확대 확률
    resolution_prob REAL,         -- 해소 확률

    -- 폴리마켓 연동
    polymarket_ids TEXT,          -- 관련 폴리마켓 이벤트 IDs

    started_at TEXT,
    last_updated TEXT,
    resolved_at TEXT
)
```

**리스크 유형별 영향 패턴:**

| 유형 | 예시 | 영향 경로 | 프리미엄 방식 |
|------|------|----------|-------------|
| **전쟁/군사 충돌** | 이란전, 우크라이나 | 유가↑, 안전자산↑, 위험자산↓ | 지속형 (해소까지 유지) |
| **정권 불안** | 탄핵, 쿠데타, 계엄 | 환율↑, 주식↓, CDS↑ | 이벤트형 + 잔류 |
| **선거/정권교체** | 미국 대선, 한국 총선 | 정책 불확실성 프리미엄 | 스텝형 (선거일까지 유지 후 해소/전환) |
| **무역전쟁** | 미중 관세, 반도체 제재 | 관련 섹터↓, 환율 변동 | 단계적 (관세 시행일 기준 스텝) |
| **제재** | 러시아 제재, 이란 제재 | 원자재↑, 관련국↓ | 지속형 (해제까지) |
| **사회 불안** | 대규모 시위, 총파업 | 해당국 자산↓, 소비↓ | 이벤트형 + 감쇠 |

**상태 전이에 따른 프리미엄 변화:**
```
escalating (확대 중)     → 프리미엄 증가, severity 상향
stable (고착)            → 프리미엄 유지, 시장 점진적 적응 (약간 감소)
de_escalating (완화 중)  → 프리미엄 감소
resolved (해소)          → 프리미엄 급감 (but 잔류 효과 1-2주)
```

**예시: 이란-이스라엘 분쟁**
```
name: "이란-이스라엘 분쟁"
status: escalating
severity: 7/10

risk_premium:
  WTI:      +8.0%  (호르무즈 해협 위험)
  gold:     +3.0%  (안전자산 수요)
  USD/KRW:  +2.0%  (원화 약세 압력)
  KOSPI:    -2.0%  (위험자산 회피)
  방산주:   +15.0% (K-방산 수혜)
  항공주:   -5.0%  (유가 상승 + 노선 우회)

scenarios:
  A. 전면전 (prob: 0.25): WTI $120+, 글로벌 위기
  B. 현상유지 (prob: 0.50): 현재 프리미엄 유지
  C. 협상 해소 (prob: 0.25): 프리미엄 해소
```

#### Layer 3: 폴리마켓 확률 (실시간 시장 합의)
폴리마켓 확률 변동이 Layer 2의 리스크 레지스터를 **자동 업데이트**.

```
폴리마켓: "이란-이스라엘 전면전" = 25%
  → escalation_prob = 0.25 (자동 연동)

확률 25% → 40% (급등)
  → severity 7→8 상향
  → risk_premium 각 항목 ×1.5 자동 조정
  → Tier 2 긴급 알림 트리거
  → 글로벌시장 + 원자재 전문가 활성화

확률 25% → 10% (급감)
  → severity 7→5 하향
  → risk_premium 각 항목 ×0.6 자동 조정
  → "해소 가능성" 알림
```

**Layer 3이 핵심인 이유:**
뉴스만으로는 "이란전이 확대될지 축소될지" 판단이 주관적.
폴리마켓은 **돈이 걸린 집단 지성**이므로 확률의 정량적 근거로 활용.

#### 3-Layer 합산 예시

```
삼성전자에 대한 총 영향 (현재 시점):

Layer 1 (이벤트):
  HBM 수율 개선 보도 (2시간 전):  +1.8% (잔여, 반감기 5일)
  외국인 순매수 (오늘):           +0.4% (잔여, 반감기 2일)

Layer 2 (지속적 리스크):
  미중 반도체 제재 (진행 중):     -1.2% (프리미엄, stable)
  한국 정치 불안 (de_escalating): -0.3% (프리미엄, 감소 중)

Layer 3 (폴리마켓):
  "미중 추가 제재" 확률 35%→30%:  제재 프리미엄 약간 완화 (+0.2%)
  "한국 조기대선" 확률 60%:       불확실성 프리미엄 유지 (0%)

총 영향: +1.8 +0.4 -1.2 -0.3 +0.2 = +0.9% 순 상승 압력
```

## 4. 분석 프레임워크

### 4.1 5계층 분석

| 계층 | 내용 | 담당 |
|------|------|------|
| **1. 레짐 판별** | Risk-On / Risk-Off / 전환기 / 위기 | Python 자동 + 거시/금융 전문가 |
| **2. 인과 사슬** | 이벤트 → 1차 → 2차 → 3차 효과 (수치화) | 관련 도메인 전문가 |
| **3. 다중 시간축** | 단기(4h-1d) + 중기(1w-1m) + 장기(3m-1y) 수렴 확인 | 리더 |
| **4. 리스크 관리** | 포지션 사이징, 상관관계, 최대 손실 한도 | 리더 + 고구마 |
| **5. 교차 검증** | 폴리마켓 + 채권 + 환율 + 원자재로 검증 | 자동 + 리더 |

### 4.2 레짐별 전략

| 레짐 | VIX | 스프레드 | 유동성 | 전략 | 사이징 |
|------|-----|---------|--------|------|--------|
| Risk-On | <18 | 축소 | 풍부 | 모멘텀, 성장주, 공격적 | 켈리 100% |
| Risk-Off | >25 | 확대 | 보통 | 평균회귀, 방어주, 현금↑ | 켈리 50% |
| 전환기 | 18-25 | 혼재 | 변동 | 소규모, 양방향 대비 | 켈리 30% |
| 위기 | >35 | 급확대 | 고갈 | 현금 극대화, 헤지 | 최소한 |

**레짐 판별 알고리즘 (Python 자동):**
```
regime_score = (
    vix_score(VIX)               × 0.35   -- VIX 수준
  + spread_score(HY_spread)      × 0.25   -- HY 스프레드 변동
  + breadth_score(advance_ratio) × 0.20   -- 시장폭
  + flow_score(safe_haven_flow)  × 0.20   -- 안전자산 자금 흐름
)

regime_score > 0.7  → Risk-On
regime_score 0.4~0.7 → 전환기
regime_score 0.15~0.4 → Risk-Off
regime_score < 0.15  → 위기
```
임계값은 `model_params` 테이블에 저장, 개선 에이전트가 조정 가능.

### 4.3 시간축별 최적 전략

| 시간축 | 전략 | 도구 |
|--------|------|------|
| 단기 (4h~1d) | 평균회귀 (과잉 반응 → 되돌림) | RSI, 볼린저, 수급 |
| 중기 (1w~1m) | 모멘텀 (오르는 것은 계속 오른다) | 이동평균, 섹터 로테이션, 실적 |
| 장기 (3m~1y) | 밸류 (싼 것이 장기 승) | P/E, FCF, 경기 사이클 |

**수렴 시 확신도 상향, 충돌 시 확신도 하향 or 관망.**

## 5. 예측 시스템

### 5.1 예측 방법론

```
예측가격(t+Δ) = 현재가 × (1 + Σ요인)

Σ요인 =
    기술적_모멘텀(Δ)           -- MA, RSI 기반 추세
  + Σ뉴스_잔여영향(t, Δ)      -- 활성 이벤트의 감쇠 합
  + 인과사슬_전파효과(t, Δ)    -- 아직 도착 안 한 2,3차 효과
  + 수급_압력(t)              -- 외국인/기관 순매수
  + 레짐_보정(t)              -- 시장 레짐에 따른 민감도
  + 계절성(t)                 -- 요일/월말/분기말 효과
  + 편향_보정                 -- 자기개선에서 발견된 체계적 편향
```

### 5.2 출력 형식 (점추정 금지)

```
삼성전자 4시간 후:
  중앙값: +1.2%
  70% 구간: [-0.3%, +2.7%]
  90% 구간: [-1.5%, +3.9%]
  확신도: 62/100

  요인 분해:
    기술적 모멘텀:   +0.5% (신뢰 0.70)
    HBM 뉴스 잔여:   +1.0% (신뢰 0.82, 반감기 잔여 72%)
    외국인 순매수:    +0.3% (신뢰 0.65)
    관세 전파:       -0.4% (신뢰 0.45, 3차)
    편향 보정:       -0.1% (강세 편향 교정)
```

### 5.3 베이지안 업데이트

기저확률(base rate)에서 시작하여 증거로 조정:
```
사전확률: "한국 주식은 60% 확률로 오른다"
  + 외국인 3일 순매수 → 65%
  + RSI 과매도 반등 → 68%
  + CPI 서프라이즈 상회 → 62%
  + 폴리마켓 관세 확률↑ → 58%
최종: 58% → 확신도 낮음 → 소규모 or 관망
```

### 5.4 실행: 켈리 기준 사이징

```
Kelly % = (승률 × 평균이익 - 패률 × 평균손실) / 평균이익
실전: Half-Kelly 사용 (과잉 베팅 방지)
```

- 확신도 비례: 확신 3/10 → 자본 2%, 확신 8/10 → 자본 10%
- 한 종목 최대 20%
- 현금 최소 15%
- 1회 매매 최대 손실 = 총자산 2%

### 6.3 고구마 (독립 개인비서)
- 팀 분석을 **절대 보지 않음** (앵커링 방지, 독립성이 가치)
- 1억원 현금으로 시작, 초기 종목 없음, 스스로 판단
- 성격: 직설적, 현실적, 독설가 (반말, "솔직히 말해서...", "이건 좀 아닌데...")
- 구체적 매매 지시 (종목, 금액, 이유)
- 매 사이클 자기 추천 결과 반성
- 벤치마크(KOSPI, S&P500) 대비 성과 추적
- **팀의 과잉분석(analysis paralysis)에 대한 해독제 역할**
- 매매 실행 가격: 사용자 수락 시점의 **실시간 시세**로 재계산 (제안 시점 가격 아님)

## 7. 보고 체계: 이벤트 구동 + 정기 하이브리드

### 7.1 3-Tier 보고

| **Tier 1: 감시** | 70초 간격, Python 자동 | 이상 탐지, 신호 품질 검증 | $0 |
| **Tier 2: 긴급 알림** | 신호 품질 > 0.7인 이상 감지 시 | 관련 전문가만 긴급 분석 + 텔레그램 알림 | Claude Code 구독 내 |
| **Tier 3: 정기 보고** | 4시간 or 시장 마감 후 | 전체 종합 분석 + 예측 + 고구마 | Claude Code 구독 내 |

### 7.2 정기 보고서 구성

```
1. 레짐 판정 (현재 시장 상태)
2. 시장 데이터 요약
3. 이전 예측 검증 (예측 vs 실제, 점수)
4. 인과 사슬 현황 (활성 이벤트 + 잔여 영향)
5. 신호 현황 (품질 통과한 신호들)
6. 새 예측 (확률 분포 + 요인 분해)
7. 에이전트 정확도 리더보드
8. 고구마 코너 (독립 분석 + 포트폴리오 + 독설)
9. 다음 예정 이벤트 (캘린더)
10. 비용/소요시간
```

### 7.3 텔레그램 인터페이스

#### 자연어 처리 (슬래시 명령어 대신)
텔레그램 대화는 **자연어로 처리**한다. 슬래시 명령어 대신 자연스러운 한국어로 소통.

```
사용자: "삼성전자 추가해줘"
봇:     "삼성전자(005930)를 감시 목록에 추가하겠습니다.
         과거 일봉 데이터(~2년)도 함께 수집합니다.
         진행할까요?"
사용자: "ㅇㅇ"
봇:     "추가 완료. 수집 시작합니다."
```

#### 2단계 의도 처리 (비용 효율)
모든 메시지에 Claude를 호출하면 비효율적. 2단계로 처리:
```
Stage 1 (Python, 즉시): 키워드/패턴 매칭으로 단순 의도 처리
  "삼전 추가" → 키워드 "추가" 감지 → 종목 추가 플로우
  "목록" → 키워드 매칭 → DB 조회 후 응답
  "ㅇㅇ", "응", "해줘" → 직전 제안 수락

Stage 2 (Claude Code, 필요 시): 복잡한 의도만 Claude에 전달
  "지금 반도체 섹터 전체적으로 어떻게 보여?" → 분석 필요 → Claude
  "고구마야, 내 포폴 너무 IT에 쏠려있지 않아?" → 고구마 에이전트
  reply "왜?" → 컨텍스트 참조 필요 → Claude
```

#### 제안→수락 패턴 (Action Confirmation)
**모든 실행 가능한 액션은 먼저 제안하고, 사용자가 수락하면 진행.**
- 종목 추가/삭제 → "~하겠습니다. 진행할까요?"
- 보고서 생성 → "전체 분석 보고서를 생성합니다. 진행할까요?"
- 고구마 매매 → "고구마가 삼성전자 500만원 매수를 제안합니다. 실행할까요?"
- 데이터 삭제/초기화 → 반드시 확인

#### 컨텍스트 유지 (Conversation Context)
텔레그램 대화의 맥락을 유지해야 한다:
- **메시지 reply 추적**: 사용자가 특정 메시지에 reply하면, 어떤 메시지(보고서, 예측, 알림 등)에 대한 반응인지 파악
- **대화 히스토리**: 최근 N개 메시지의 맥락 유지 (DB 저장)
- **참조 해석**: "이거 왜 이래?" → 직전 보고서/알림의 내용을 참조
- **후속 질문**: "더 자세히" → 직전 분석의 상세 버전 제공
- **고구마 대화**: 고구마에게 직접 질문 가능 ("고구마야, 지금 삼전 어떻게 봐?")

```
봇:     "[알림] 삼성전자 +3.2% 급등 (신호 품질 0.82)"
사용자: (reply) "왜?"
봇:     "HBM3E 수율 개선 보도 (30분 전, 잔여 영향 +2.1%)
         + 외국인 순매수 1,200억 (수급 +0.8%)
         + KOSPI 전체 상승 기조 (시장폭 62%)
         인과 사슬 2차 효과: SK하이닉스 연동 상승 예상 (+1.5%, 신뢰 0.65)"
```

#### 지원하는 자연어 의도
| 의도 | 예시 표현 |
|------|----------|
| 종목 추가 | "삼성전자 추가", "NVDA 봐줘", "비트코인도 추적해" |
| 종목 삭제 | "삼전 빼줘", "TSLA 관심 없어" |
| 목록 조회 | "지금 뭐 보고 있어?", "감시 목록" |
| 보고서 | "보고서 줘", "지금 상황 어때?", "분석해줘" |
| 예측 | "삼전 어떻게 될 것 같아?", "내일 전망은?" |
| 정확도 | "누가 제일 잘 맞춰?", "적중률 보여줘" |
| 포트폴리오 | "고구마 포폴", "수익률 어때?" |
| 인과 사슬 | "삼전 왜 올라?", "이거 뭐 때문이야?" |
| 신호 | "지금 신호 있어?", "뭔가 특이한 거 있어?" |

## 8. 자기개선 시스템

### 8.1 목표
**"보정(Calibration) 극대화"**

60% 확신 예측이 실제로 60% 맞으면 **완벽한 보정**.
이것이 정확도보다 중요 — 보정이 맞으면 켈리 기준이 최적 작동 → 수익 극대화.

### 8.2 보정 대상 (5가지)

| 보정 대상 | 설명 | 방법 |
|-----------|------|------|
| **반감기 추정** | 뉴스 효과가 예상보다 빨리/느리게 소멸 | 실측 대비 보정계수 |
| **충격 크기 추정** | 이벤트 영향이 과대/과소 추정 | 실측 대비 보정계수 |
| **신호 품질 보정** | 신호 필터의 임계값 조정 | was_real 필드 vs 품질 점수 |
| **에이전트 편향** | 특정 에이전트의 체계적 강세/약세 편향 | 방향별 오차 누적 분석 |
| **레짐별 민감도** | 레짐에 따라 전략 효과가 다름 | 레짐별 정확도 분리 추적 |

### 8.3 메커니즘

```
매 보고 사이클:
  1. 만료된 예측 검증 (자동)
  2. 보정 지표 업데이트 (자동)
  3. 임계치 미달 감지 (방향 적중률 < 55% or 보정 오차 > 15%)
     → Claude Code 서브에이전트로 실패 패턴 분석
     → 전략 노트 생성/갱신
  4. 전략 노트를 다음 예측 프롬프트에 주입
```

### 8.4 전략 노트 생명주기

```sql
strategy_notes (
    agent_role,
    content,
    valid_regime,           -- 'risk_on', 'risk_off', 'all'
    expires_at,             -- 자동 만료 (기본 30일)
    decay_confidence,       -- 시간 경과 시 신뢰도 감소
    contradicted_count,     -- 반례 횟수
    active                  -- contradicted_count >= 3이면 자동 비활성화
)
```

### 8.5 에이전트 가중치 동적 조정

앙상블 시 정확한 에이전트에 높은 가중치:
```
에이전트 가중치 = 기본(1/N) × 정확도 보정 × 레짐 보정

예: 현재 레짐 risk-off일 때
  거시 전문가: 기본 0.14 × 적중률 0.72 × 레짐 1.3 = 0.131 (높은 가중치)
  미시 전문가: 기본 0.14 × 적중률 0.65 × 레짐 0.8 = 0.073 (낮은 가중치)
```

## 9. 아키텍처: Python + Claude Code (API 아님)

### 9.1 핵심 원칙
> **Claude API(anthropic SDK)를 사용하지 않는다. Claude Code를 직접 사용한다.**

- Python: 24/7 상주, 데이터 수집, 신호 감지, DB, 텔레그램 봇
- Claude Code: 분석, 예측, 인과 추적, 자기개선 — **서브에이전트 + MCP + 스케줄 트리거**로 실행
- anthropic Python SDK 호출 없음. 대신 Claude Code CLI(`claude`) 또는 내장 Agent 시스템 사용
- 에이전트가 MCP를 통해 능동적으로 데이터 조회 (수동적 프롬프트 주입 탈피)

**왜 Claude Code 직접 사용?**
- API 과금 대신 Claude Code 구독으로 비용 예측 가능
- 서브에이전트, Extended Thinking, WebSearch 등 네이티브 기능 그대로 활용
- MCP 서버 연동이 자연스러움 (별도 클라이언트 구현 불필요)
- 스케줄 트리거로 cron 없이 자동 실행

### 9.2 Python 상주 프로세스

| 역할 | 세부 |
|------|------|
| 데이터 수집 | 70초 간격 (주식/인덱스), 5분 간격 (암호화폐/폴리마켓/원자재/채권) |
| 이상 탐지 | 가격/거래량 급변 자동 감지 |
| 신호 품질 검증 | 7단계 필터 (자동 계산) |
| 뉴스 잔여영향 계산 | 감쇠 함수 실시간 적용 |
| 기술 지표 계산 | SMA, RSI, MACD, 볼린저, 피보나치 등 |
| DB 관리 | SQLite WAL 모드 |
| 텔레그램 봇 | 명령어 처리, 메시지 전송 |
| MCP 서버 | Claude 에이전트에 금융 데이터 도구 노출 |
| 이벤트 캘린더 | D-7/3/1/0 알림 + 에이전트 활성화 트리거 |

### 9.5 에이전트 간 데이터 흐름

```
보고 사이클 실행 흐름:

1. Python: 현재 시장 데이터 + 신호 + 인과 사슬 상태를 DB에 준비
2. 스케줄 트리거 → 리더 에이전트 시작
3. 리더: MCP로 레짐 판정 데이터 조회 → 활성화할 전문가 결정
4. 리더: 선택된 전문가 서브에이전트를 병렬 실행
   ├─ 각 전문가: MCP로 자기 도메인 데이터 조회
   ├─ 각 전문가: WebSearch로 실시간 뉴스 검색
   └─ 각 전문가: 분석 결과를 텍스트로 반환
5. 리더: 전문가들의 분석 결과를 수신 → 종합 판단 (Extended Thinking)
6. 고구마: 리더와 병렬로 독립 실행 (MCP만 사용, 전문가 결과 안 봄)
7. 리더 + 고구마 결과 → Python이 보고서 조립 → 텔레그램 전송
8. Python: 예측을 DB에 저장, 이전 예측 검증
```

전문가 → 리더 전달: **서브에이전트 반환값 (텍스트)**. 별도 DB/파일 불필요.
리더가 전문가를 서브에이전트로 호출하면, 결과가 리더의 컨텍스트에 직접 포함.

| **서브에이전트** | 전문가 팀 관리, 독립 실행 |
| **스케줄 트리거** | 정기 보고, 자기개선 사이클 |
| **WebSearch** | 에이전트가 실시간 뉴스/맥락 검색 |
| **Extended Thinking** | 리더 종합 판단, 고구마 포트폴리오 결정 |
| **Hooks** | 포트폴리오 변경 시 자동 검증 |

### 10.3 운영 한도

- 조건부 활성화로 불필요한 에이전트 실행 방지 (비용이 아닌 효율 목적)
- Claude Code 사용량 한도에 도달 시: 조건부 전문가 수 축소 (모델 다운그레이드 아님)

## 11. 시스템 운용팀

### 11.1 역할: 시스템 자체를 개선하고 관리하는 팀

```
투자 분석팀: "시장을 분석한다" → 전략 노트 (DB) → 즉시 반영
시스템 운용팀: "시스템을 개선한다" → 코드 변경 (Git) → 검증 후 배포
```

### 11.2 팀 구성

| 역할 | 담당 | 모델 | 트리거 |
|------|------|------|--------|
| **개선 에이전트** | 성능 분석 → 코드/프롬프트 수정 | Sonnet | 일 1회 or 정확도 급락 시 |
| **QA 에이전트** | 변경사항 검증 (구문, 로직, 회귀) | Sonnet | 개선 에이전트 작업 완료 시 |
| **DevOps 에이전트** | Git 관리, 배포, 롤백, DB 관리 | Sonnet | QA 통과 + 사용자 승인 시 |

### 11.3 자기개선 2레벨

```
Level 1: 전략 조정 (분석팀, 매 사이클)
  → 전략 노트 DB → 다음 프롬프트에 주입
  → 코드 변경 없음, 즉시 반영
  예: "삼전에 강세 편향 -0.3% 보정"

Level 2: 시스템 개선 (운용팀, 일 1회)
  → 코드/프롬프트/파라미터 실제 변경
  → Git 브랜치 → QA 검증 → 사용자 승인 → 머지 → 배포
  예: "신호 필터 임계값 0.7→0.65로 조정"
  예: "한국시장 전문가 프롬프트에 수급 분석 강화"
```

### 11.4 개선 에이전트 워크플로우

**입력 데이터:**
- 예측 정확도 (에이전트별, 레짐별, 종목별)
- 신호 필터 성능 (`was_real` vs `final_quality`)
- 충격 크기/반감기 보정계수 추이
- 에러 로그 (최근 3일)
- DB 성장률, 쿼리 성능

**변경 가능 범위:**
- ✅ 에이전트 프롬프트 (`.claude/agents/*.md`)
- ✅ 신호 필터 파라미터
- ✅ 기술 지표 파라미터 (RSI 기간, 볼린저 σ 등)
- ✅ 반감기/충격 크기 기본값
- ✅ 레짐 판별 임계값 (VIX 18/25/35)
- ✅ 이상 탐지 임계값
- ✅ 텔레그램 봇 응답 로직
- ❌ DB 스키마 (위험, 수동만)
- ❌ MCP 서버 코어 (위험)
- ❌ `.env`, 인증 정보 (보안)

**최대 3개 개선/사이클.** 과잉 변경 방지.

### 11.5 Git 워크플로우 + 무인 운영

**원칙: 시스템 개선, 장애 복구, 모니터링은 무인. 매매 판단만 사용자 개입.**

```
1. improve/YYYYMMDD 브랜치 생성
2. 코드 변경 (최대 3건)
3. QA 에이전트 검증:
   - Python 구문 검사 (py_compile)
   - Shell 구문 검사 (bash -n)
   - 기존 테스트 실행
   - 변경 전후 로직 비교
4. QA 통과 → 자동 머지 → 자동 배포 → 텔레그램 사후 알림
   QA 실패 → 브랜치 삭제 → 텔레그램 실패 알림
```

**무인 자동 처리 (사용자 개입 불필요):**
- 시스템 개선: QA 통과하면 자동 머지 + 배포 (사후 알림만)
- 장애 복구: 크래시 → systemd 자동 재시작
- API 장애: fallback 자동 전환 → 복구 시 자동 복원
- DB 관리: 백업, 압축, 무결성 검사 전부 자동
- 배포 실패: 자동 롤백 → 이전 버전 복원
- 모니터링: 이상 감지 시 텔레그램 알림 (조치는 자동)

**사용자 개입이 필요한 것 (매매/투자 판단만):**
- 종목 추가/삭제
- 고구마 매매 실행 승인
- 보고서 기반 투자 결정
- DB 스키마 변경 (드문 경우)

### 11.6 DevOps 에이전트: DB 관리

**일상 운영:**
- 일일 DB 백업 (압축, 7일 보관)
- DB `PRAGMA integrity_check` (시작 시 + 일 1회)
- 90일 이전 분봉 데이터 압축 (일봉으로 집계 후 원본 삭제)
- 일봉 데이터는 영구 보관

**DB 마이그레이션:**
- 스키마 변경이 필요할 때 (개선 에이전트가 제안 불가, DevOps만 가능)
- 마이그레이션 SQL 생성 → 백업 → 적용 → 검증
- 실패 시 백업에서 자동 롤백

**DB 모니터링:**
- 테이블별 레코드 수, 용량 추적
- 슬로우 쿼리 감지 (100ms 이상)
- 인덱스 효율성 점검 (월 1회)
- WAL 파일 크기 모니터링 → 비정상 증가 시 알림

**백업 전략:**
```
일일 백업:    invest.db → backups/invest_YYYYMMDD.db.gz (7일 보관)
주간 백업:    invest.db → backups/weekly/invest_YYYYMMDD.db.gz (4주 보관)
마이그레이션: invest.db → backups/pre_migration_YYYYMMDD.db (수동 삭제)
```

**텔레그램 보고 (주 1회):**
```
봇: "📦 DB 상태 보고
     크기: 142MB
     레코드: stock_prices 1.2M, predictions 3.4K, events 580
     이번 주 성장: +18MB (+14%)
     백업: 7/7 성공
     무결성: ✅ 정상
     슬로우 쿼리: 0건
     압축 대상: 분봉 28일분 → 압축할까요?"
```

### 11.7 배포/롤백 (무인)

```
배포: main 머지 → systemd 서비스 재시작 → 헬스체크 (30초)
정상: 텔레그램 "✅ 배포 완료 (변경 내용 요약)"
실패: 자동 롤백 (git revert) → 이전 버전 재시작 → 텔레그램 "❌ 배포 실패, 자동 롤백 완료"
```

### 11.8 무인 운영 요약

```
사용자가 하는 일:
  - 텔레그램에서 보고서 읽기
  - 종목 추가/삭제 지시
  - 고구마 매매 승인/거부
  - (가끔) 시스템 알림 확인

시스템이 알아서 하는 일:
  - 데이터 수집 (70초/5분 간격, 24/7)
  - 이상 탐지 + 신호 필터링
  - 에이전트 분석 + 예측 + 보고서 생성
  - 예측 검증 + 보정계수 업데이트
  - 시스템 개선 (코드 변경 → QA → 자동 배포)
  - DB 백업, 압축, 무결성 검사
  - 장애 감지 → 자동 복구 → 사후 알림
  - Git 브랜치 관리, 머지, 배포, 롤백
```

### 11.9 장애 자동 복구

**Python 프로세스:**
- systemd 서비스 (`Restart=always`)
- 헬스체크: 마지막 수집 시각 5분 초과 → 텔레그램 알림
- 로그: `logs/`, 일별 로테이션

**API 장애:**
- 네이버 장애/차단 → pykrx/yfinance fallback 자동 전환
- fallback도 실패 → 마지막 성공 데이터 유지 + 텔레그램 알림
- 5회 연속 실패 → 해당 소스 30분 쿨다운

**시스템 상태 (텔레그램 "시스템 상태" 질문으로 확인):**
- 마지막 수집 시각 (소스별)
- DB 크기, 레코드 수, 백업 상태
- 활성 에이전트, 마지막 보고서 시각
- API 오류 카운트 (24시간)
- 최근 개선/배포 이력

## 12. 결정된 사항
- ✅ 언어: Python (AI/금융 라이브러리 풍부)
- ✅ 실행: Claude Code 직접 사용 (API 아님), Max 구독
- ✅ 데이터 소스: 네이버 API 최우선, yfinance는 fallback만
- ✅ 고구마: 1억 현금, 초기 종목 없음, 스스로 판단
- ✅ 보고 체계: 이벤트 구동 + 정기 하이브리드 (3-Tier)
- ✅ 에이전트: 분석팀(9명) + 운용팀(3명), 조건부 활성화, 최소 Sonnet
- ✅ 예측: 확률 분포 출력, 점추정 금지, 엣지 없으면 패스
- ✅ 충격 모델: 5요인 곱연산 (Base × Surprise × Regime × Positioning × Liquidity)
- ✅ 3-Layer 모델: 이벤트 반감기 + 지정학 리스크 레지스터 + 폴리마켓 확률
- ✅ 자기개선: Level 1 전략 노트(즉시) + Level 2 코드 개선(무인 배포)
- ✅ 무인 운영: 시스템 개선/장애복구/모니터링은 자동, 매매 판단만 수동
- ✅ 텔레그램: 자연어 처리, 2단계 의도 분류, 제안→수락, 컨텍스트 유지
- ✅ 비트코인: 네이버 `front-api/crypto/top` (KRW + 김치프리미엄)
- ✅ 뉴스 반감기: 레짐별 차등, 감쇠 형태 4종, 자기개선 보정
- ✅ 파라미터: DB `model_params` 테이블 (코드 변경 없이 조정 가능)
- ✅ 보고서 언어: 한국어
